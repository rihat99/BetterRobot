# 10 · User-facing docs: Sphinx + MyST + Diátaxis

★★ **Structural.** Adds a documentation site without doubling the
maintenance footprint. References the `sphinx-docs` and
`diataxis-docs` skills already configured in this workspace.

## Problem

`docs/` today is *internal*: design specs, conventions, status. It is
not a place a new user can land and learn how to drive the library.
A user typing `import better_robot as br` and looking for a tutorial
finds:

- A README pointing at design docs with names like
  `02_DATA_MODEL.md` and `13_NAMING.md`.
- No tutorial. No "how to compute FK on Panda in 5 lines."
- No auto-generated reference. Every public symbol's docstring is
  there, but there's no page that lists them.

For the long-term goal — *library used widely by community for years*
— this is a real blocker. Adopters don't read internal specs first;
they look at the homepage, paste an example, and judge in 60 seconds
whether the library is for them.

The bar is not a hand-written 500-page manual. The bar is:
auto-generated reference + a small handful of well-curated
tutorials. Sphinx + MyST + the
[Diátaxis](https://diataxis.fr/) framework gives us that with low
maintenance.

## The proposal

### 10.A Adopt Diátaxis

Diátaxis splits docs into four orthogonal quadrants:

| Quadrant | Goal | Example |
|----------|------|---------|
| **Tutorials** | Learning | "Solve an IK problem on Panda in 10 minutes" |
| **How-to guides** | Tasks | "How to add a custom residual" |
| **Reference** | Information | Auto-generated API reference per symbol |
| **Concepts** (Explanation) | Understanding | "Why frozen `Model` + mutable `Data`" |

Each page belongs to **exactly one** quadrant. Mixing them is the
default failure mode.

the contract test in [Proposal 06](06_public_api_audit.md) ensures
every public symbol has one.

### 10.C Tooling

- **Sphinx** with the **Furo** theme (clean, dark-mode-aware, low
  maintenance).
- **MyST-Parser** for Markdown.
- **MyST-NB** for executable notebooks (run in CI; broken examples
  fail the build — this is the existing
  [16 §7 Example-as-test](../../conventions/16_TESTING.md) discipline
  applied to docs as well).
- **sphinx-autodoc + sphinx-autosummary** for the API reference.
- **sphinx.ext.napoleon** for the docstring style. The drafts under
  `style/` disagree on Google vs NumPy; pick **NumPy-style** —
  see [Proposal 13](13_style_reconciliation.md) — to match the
  scientific-Python ecosystem norm.
- **sphinx_design** for tabbed examples (PyTorch vs NumPy when
  comparing) and grid layouts on the landing page.

### 10.D What the landing page shows

Three things, in order:

1. A 5-line `solve_ik` snippet that runs (taken from
   `examples/01_basic_ik.py`).
2. A grid of "What can BetterRobot do?" cards: FK, IK, Jacobians,
   visualisation, batched on GPU.
3. Three buttons: "Tutorial" / "How-to" / "Reference" / "Concepts".

Not a wall of text. A user has 60 seconds.

### 10.E Tutorials we ship in v1

Four tutorials cover the headline use cases:

1. **Install + Panda FK.** Loads Panda, runs FK at neutral, reads
   the hand pose. Demonstrates `load`, `forward_kinematics`,
   `Data.frame_pose`. **15 min.**
2. **Basic IK on Panda.** Sets a target hand pose, calls `solve_ik`,
   visualises the result. Demonstrates `IKCostConfig`,
   `OptimizerConfig`, `IKResult.fk()`. **20 min.**
3. **Floating-base IK on G1.** Same as 2 with `free_flyer=True`.
   Demonstrates that the API is uniform across base topologies.
   **20 min.**
4. **Custom residual.** Walks through writing a
   `MinTorqueResidual` against the existing FK, registering it,
   adding it to a `CostStack`. **45 min.**

Tutorials live in `docs/site/tutorials/` as `.md` files with
embedded MyST-NB code cells. CI runs them headless on every build.

### 10.F How-to guides

Short, recipe-shaped pages — no narrative, no learning objective:

- "How to load a URDF with a free-flyer base"
- "How to batch IK across GPU device with `B=1024`"
- "How to add a new joint kind"
- "How to add a new optimizer"
- "How to plug PyTorch's CUDA Graphs into the IK loop"
- "How to debug analytic-vs-autodiff Jacobian mismatch"

Each page is one screen long. Each has a code block at the top, prose
below it.

### 10.G Concepts pages

The hard part. These are the explanations that make Sphinx a real
reference:

- **Model and Data** — why frozen, why mutable, when to clone.
- **Joint taxonomy** — the universal joint system, why no fixed/
  floating split.
- **Jacobian strategies** — what `ANALYTIC`, `AUTODIFF`,
  `FUNCTIONAL`, `FINITE_DIFF`, `AUTO` mean and when to use each.
- **Lie and spatial** — the typed/raw split, why no operator
  overloading on `Motion`/`Force`.
- **Batched by default** — the `(B, [T,] ..., D)` convention.
- **Extension seams** — pointer to
  [15_EXTENSION.md](../../conventions/15_EXTENSION.md), but
  reformatted as a concept guide for users (rather than a contract
  for contributors).

Each is ~500 words plus a diagram. They are written once and rarely
change — they're explanatory, not state-of-the-art.

### 10.H Internal vs user docs — the split

| Folder | Audience | Tooling |
|--------|----------|---------|
| `docs/design/`, `docs/conventions/`, `docs/status/`, `docs/style/`, `docs/claude_plan/` | Library contributors | Plain Markdown, read in editor |
| `docs/site/` | Library users | Sphinx-built HTML |

The two never cross-reference each other in the user-facing site
output, but contributors can link freely from `docs/design/` to
`docs/site/concepts/` to keep narrative consistent.

### 10.I CI integration

- Build the docs in CI: `sphinx-build -b html docs/site/ build/html/`.
- Run notebook tutorials with `MyST-NB` execution mode `force`.
- Treat broken cross-references and missing references as build
  errors (`-W` flag).
- Deploy to GitHub Pages (or similar) on merge to main.

A single `make docs` target in `pyproject.toml` (or a
`uv run task docs` recipe) runs all of this locally.

### 10.J Apidoc generation strategy

Don't use `sphinx-apidoc` (it produces ugly stub `.rst` files).
Use **`autodoc2`** (or `sphinx.ext.autodoc` + `autosummary` with
`generate=True`):

```rst
# docs/site/reference/api/index.md

# API reference

The complete public API of `better_robot`.

```{autosummary}
:toctree: ./
:recursive:

better_robot.Model
better_robot.Data
better_robot.SE3
better_robot.SO3
better_robot.forward_kinematics
better_robot.solve_ik
...
```

The list is auto-generated from `better_robot.__all__`. A docstring
contract test ([Proposal 06 §6.E](06_public_api_audit.md)) ensures
each entry has a usable example.

## Tradeoffs

| For | Against |
|-----|---------|
| Users have a real on-ramp. The 60-second test passes. | Adds Sphinx + MyST + Furo + sphinx-design + autodoc2 to the dev dep tree. Mitigation: under `[docs]` extras only. |
| Writing tutorials forces us to *use* the API the way users will, surfacing rough edges before users do. | Tutorials need ongoing maintenance as the API evolves. Mitigation: notebooks run in CI; rot is caught. |
| The split keeps internal specs internal — they don't bleed into the user manual. | Two doc trees to navigate. Mitigation: `docs/README.md` explains the split. |

## Acceptance criteria

- `docs/site/` exists with the layout above.
- `pyproject.toml` declares a `[docs]` extra with all build deps.
- `sphinx-build -W -b html docs/site/ build/html/` succeeds without
  warnings.
- The four v1 tutorials run end-to-end in CI under MyST-NB.
- `tests/contract/test_examples.py` is extended to cover any
  notebook in `docs/site/tutorials/` so docs and examples share the
  same regression net.
- The auto-generated API reference contains every symbol in
  `better_robot.__all__` and links to its source.
